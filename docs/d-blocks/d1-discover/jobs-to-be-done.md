# Jobs-to-be-Done Analysis

## Обзор

Jobs-to-be-Done (JTBD) — это фреймворк, который фокусируется на понимании основных "работ", для выполнения которых клиенты "нанимают" ваш продукт. Этот подход, популяризированный Клейтоном Кристенсеном, смещает фокус с демографических характеристик клиентов на их фундаментальные потребности и цели.

&gt; "Маркетинг не о продуктах, которые вы создаете, а об историях, которые вы рассказываете." — Клейтон Кристенсен

## Ключевые диагностические вопросы

1. **Какую основную "работу" наши клиенты нанимают наш продукт выполнить?**
2. **Как клиенты решали эту задачу ДО нашего продукта?**
3. **В каком контексте возникает потребность в этой "работе"?**

## Метрики и формулы

### Балл ясности Jobs-to-be-Done

```
Балл Ясности Jobs-to-be-Done = (Четко Определенные Работы ÷ Общее Количество Сегментов) × 100
```

**Логика расчета:**
Измеряет, для скольких клиентских сегментов вы четко понимаете основную работу-которую-нужно-выполнить.

**Эталоны:**
- "100 процентов" ясность = исключительное понимание клиентов
- "80 процентов" и более ясность = хорошее определение работы
- 60-"80 процентов" ясность = умеренное понимание
- "Менее чем 60 процентов" ясность = риск путаницы в работе

**Пример:**
SaaS компания с 5 клиентскими сегментами четко определила основную работу для 4 сегментов.
Балл Ясности = (4 ÷ 5) × 100 = 80% (хорошо!)

### Балл возможности Jobs

```
Балл Возможности Jobs = Рейтинг Важности - Рейтинг Удовлетворенности
```

**Логика расчета:**
Для каждой работы: важность минус текущая удовлетворенность. Показывает наибольшие возможности для улучшения.

**Эталоны:**
- "Более чем 7 баллов" = массивная возможность (приоритизировать немедленно)
- 5-7 баллов = значительная возможность (приоритет roadmap)
- 3-5 баллов = умеренная возможность (рассмотреть)
- "Менее чем 3 баллов" = насыщенная работа (избегать)

**Пример:**
Работа: "Автоматизация финансовой отчетности"
Важность: 9/10, Текущая удовлетворенность: 3/10
Балл Возможности = 9 - 3 = 6 баллов (значительная возможность!)

### Балл конкурентного замещения

```
Балл Конкурентного Замещения = (Принятие Нашего Решения ÷ Использование Предыдущего Решения) × 100
```

**Логика расчета:**
Процент клиентов, которые полностью заменили старое решение нашим продуктом.

**Эталоны:**
- "Более чем 80 процентов" замещение = ценность уровня монополии
- 60-80 процентов замещение = сильное ценностное предложение
- 40-60 процентов замещение = умеренное улучшение
- "Менее чем 40 процентов" замещение = инкрементальная ценность

**Пример:**
100 клиентов использовали Excel для задач, 85 клиентов полностью перешли на наш продукт.
Балл Замещения = (85 ÷ 100) × 100 = 85% (отлично!)

### Предсказуемость контекстных триггеров

```
Предсказуемость Контекстных Триггеров = (Предсказанные Триггеры ÷ Фактические Триггеры Покупки) × 100
```

**Логика расчета:**
Точность прогнозирования триггеров покупки на основе контекстуальных паттернов.

**Эталоны:**
- "Более чем 75 процентов" предсказуемость = исключительное понимание контекста
- 60-75 процентов предсказуемость = хорошее осознание триггеров
- 45-60 процентов предсказуемость = умеренное прогнозирование
- "Менее чем 45 процентов" предсказуемость = реактивный маркетинг

**Пример:**
Предсказали 20 триггеров покупки для корпоративного сегмента, 18 триггеров точно предсказали фактические покупки.
Предсказуемость Триггеров = (18 ÷ 20) × 100 = 90% (исключительно!)

## Методология исследования JTBD

### 1. Выявление работ

- **Интервью с клиентами**: Фокус на моменты "переключения" (когда клиент решил использовать ваш продукт)
- **Наблюдение за использованием**: Как клиенты фактически используют продукт
- **Анализ альтернатив**: Что клиенты использовали раньше и почему переключились

### 2. Приоритизация работ

- **Оценка важности**: Насколько критична работа для клиента
- **Оценка удовлетворенности**: Насколько хорошо текущие решения выполняют работу
- **Расчет возможности**: Важность минус удовлетворенность

### 3. Детализация работ

- **Функциональные аспекты**: Практическая задача, которую нужно выполнить
- **Эмоциональные аспекты**: Как клиент хочет себя чувствовать
- **Социальные аспекты**: Как клиент хочет восприниматься другими

### 4. Контекстуальный анализ

- **Ситуационные триггеры**: Когда возникает потребность в работе
- **Барьеры использования**: Что мешает выполнению работы
- **Критерии успеха**: Как клиент понимает, что работа выполнена успешно

## Руководство по развитию

### Базовый уровень (от 1 до 3)
- Проведите интервью Jobs-to-be-Done с каждым сегментом
- Создайте базовую карту работ для основных сегментов
- Определите 1-2 ключевые работы для фокуса продукта

### Средний уровень (от 4 до 6)
- Валидируйте работы через данные поведения клиентов
- Создайте детальную иерархию работ (основные, вспомогательные)
- Проанализируйте конкурентное замещение для каждой работы

### Продвинутый уровень (от 7 до 9)
- Используйте JTBD для позиционирования стратегии D2
- Создайте прогнозные модели контекстуальных триггеров
- Интегрируйте JTBD в процесс разработки продукта

## Интеграция с другими D-блоками

- **D2 DEFINE**: JTBD формирует основу для бизнес-стратегии и позиционирования
- **D3 DESIGN**: Дизайн пользовательского опыта оптимизируется для выполнения ключевых работ
- **D5 DEPLOY**: Маркетинговые сообщения фокусируются на ключевых работах
- **D6 DELIVER**: Успех клиентов измеряется через выполнение работ
- **D7 DATA**: Аналитика отслеживает эффективность выполнения работ

## Шаблоны и инструменты

### Шаблон интервью JTBD

```
1. Расскажите о последнем случае, когда вы использовали [продукт].
2. Что вы пытались достичь?
3. Как вы решали эту задачу раньше?
4. Что побудило вас искать новое решение?
5. Какие альтернативы вы рассматривали?
6. Почему вы выбрали именно наш продукт?
7. Как вы поняли, что задача успешно выполнена?
```

```
Коэффициент Концентрации Выручки = (Выручка Топ-20% Клиентов ÷ Общая Выручка) × 100
```

**Логика расчета:**
Процент выручки

## 📊 Скоринг ясности (Clarity Scores)

При оценке ясности задач используйте следующую шкалу:

- "Менее 4" = Нечёткая задача (требуется уточнение)
- 4-7 = Средняя ясность (можно приступать к работе)
- 8-10 = Кристально ясная задача (оптимально для разработки)

После создания списка работ проведите подсчет среднего балла ясности. Команда должна стремиться к показателю **не ниже 7.5**.

# Jobs to be Done

## Оценка ясности целей и решаемых проблем (JTBD Score)

JTBD Score измеряет насколько четко и точно определены задачи, которые пользователь пытается решить с помощью вашего продукта.

### Как считается:

Оценка выводится по шкале от 1 до 10, где:

- 1.0-3.0 = минимальное понимание задач пользователя
- 3.1-5.0 = базовое понимание основных задач 
- 5.1-7.0 = уверенное понимание ключевых задач и контекстов
- 7.1-9.0 = глубокое понимание всего спектра задач пользователя
- 9.1-10 = полное понимание всех задач, контекстов и блокеров

### Ключевые факторы для оценки:

#### Оценка задач пользователя:

- **Функциональные аспекты**: Что конкретно пользователь пытается сделать?
- **Эмоциональные аспекты**: Какие чувства пользователь хочет испытать или избежать?
- **Социальные аспекты**: Как задача связана с отношениями и социальным статусом?

#### Оценка контекста выполнения задачи:

- **Когда** пользователь пытается выполнить задачу?
- **Где** пользователь находится в момент выполнения задачи?
- **Почему** пользователь выбирает именно этот момент?

#### Оценка предсказуемости контекстуальных триггеров:

- Менее 3.0 = триггеры случайны и непредсказуемы
- 3.1-5.0 = основные триггеры поняты, но их появление сложно предсказать

## Оценка возможностей (Opportunity Score)

Opportunity Score отражает величину возможности для вашего продукта на рынке.

### Формула:
**Opportunity Score = Importance + (Importance - Satisfaction)**

Где:
- **Importance**: важность задачи для пользователя (от 1 до 10)
- **Satisfaction**: текущая удовлетворенность существующими решениями (от 1 до 10)

### Интерпретация результатов:

- "Менее 5" = слабая возможность
- 5-10 = средняя возможность
- 10-15 = сильная возможность
- "Более 15" = исключительная возможность

### Пример расчета:

Задача "Быстро найти нужный документ в большом архиве":
- Importance = 9 (очень важно)
- Satisfaction = 3 (плохо решается существующими продуктами)
- Opportunity Score = 9 + (9 - 3) = 9 + 6 = 15

## Оценка конкурентного замещения (Competitive Replacement Score)

Competitive Replacement Score показывает, насколько ваш продукт способен заменить существующие решения.

### Как считается:
Оценка выводится по шкале от 1 до 10, где:

- 1.0-3.0 = продукту сложно конкурировать с существующими решениями
- 3.1-5.0 = продукт предлагает некоторые преимущества перед конкурентами
- 5.1-7.0 = продукт значительно лучше решает задачи пользователя
- 7.1-9.0 = продукт радикально меняет способ решения задачи
- 9.1-10 = продукт создает новую категорию, делая существующие решения неактуальными

### Ключевые факторы для оценки:

1. **Переключаемость** — насколько легко пользователю перейти на ваше решение?
2. **Превосходство** — насколько ваше решение превосходит существующие?
3. **Привычки** — насколько глубоко укоренены привычки использования текущих решений?
4. **Экосистема** — насколько ваш продукт интегрируется в существующую экосистему пользователя?

## Методология исследования и разработки JTBD

### ⚠️ Обязательно используйте следующие методы:

1. **Интервью "5 почему"** — глубинные интервью для выявления истинных мотивов использования продукта
2. **Анализ "моментов переключения"** — изучение ситуаций, когда пользователь меняет одно решение на другое
3. **Карты задач пользователя** — визуализация всех задач, которые пользователь пытается решить
4. **Сегментация по задачам** — группировка пользователей не по демографии, а по схожести решаемых задач
5. **Push/Pull факторы** — анализ, что "отталкивает" пользователей от старых решений и "притягивает" к новым

**Социальный аспект**: Получить одобрение и признание аудитории

**Контекстуальные триггеры:**
- Смена работы/карьерного пути
- Ребрендинг личного бренда
- Сезонное обновление контента

**Opportunity Score:**
- Importance = 8
- Satisfaction = 4
- Opportunity Score = 8 + (8 - 4) = 12 (сильная возможность)
